# AI 虚拟小镇 V2 项目说明

更新时间：2026-06-10

## 项目定位

AI 虚拟小镇 V2 是一个事件驱动、多 Agent、可持久化的虚拟小镇 Demo。目标不是让 AI 随机讲故事，而是把很多个 AI 角色放进同一个“盒子”里，让他们根据时间、地点、天气、家庭、关系、记忆、需求、情绪、承诺、长期目标和知识边界持续生活。

当前版本采用前端持有世界状态、后端负责云 AI 调用和配置/存档文件读写的结构。

- 前端页面：`ai-town-v2.html`
- 后端服务：`ai-town-v2-server.js`
- 配置文件：`ai-town-config.json`
- 存档目录：`saves/`
- 访问地址：`http://localhost:8788/`

## 核心目标

每个角色应该像一个“活在镇里的人”，而不是一次性对话机器人。角色会持续携带自己的状态，并被世界规则约束。

每个角色包含：

- 当前位置、移动状态
- 职业、固定日程、动态日程
- 真实年龄：`ageYears` / `ageDays` / `ageStage`
- 家庭、邻里、关系矩阵
- 多维情绪
- 需求状态条
- 情绪到行为的调制
- 分层记忆
- 知识边界
- 承诺/任务债务
- 长期目标和自我叙事
- 事件队列
- 生命状态：存活、濒危、死亡

小镇本身包含：

- 地点 Agent
- 日期、农历、节气、天气
- 地点工作人员约束
- 公共事件
- 知识传播
- 关系事件链
- 每日 0 点结构结算
- 存档管理和配置管理

## 当前运行逻辑

每轮模拟不是让所有角色一起行动，而是按顺序执行：

1. WeatherAgent：按需生成天气观测、6 小时报告和 7 天趋势。
2. LocationEventAgent：按时间段生成地点内部小事件。
3. LocationDailyAgent / LocationChainAgent：生成地点今日重点，并维护地点连续事件链。
4. LocationRuntimeAgent：判断地点此刻是否开放、拥挤、缺岗、排队或受事件链影响。
5. ProcessManagerAgent：优先检查未完成的跨回合行动过程。
6. Scheduler：判断哪些角色当前值得行动。
7. AgentAction：为被选中的角色生成小行动。
8. TimePassageAgent：判断本轮虚拟时间内行动是否完成、是否留下跨回合过程。
9. StateSettlementAgent：根据已发生行动提出状态补丁建议。
10. WorldGuard 本地审查：修正越界知识、时间地点身份冲突、隐形工作人员、全镇广播、未完成行动写成已完成等问题。
11. EventImpactAgent / InformationPropagationAgent / SocialProcessAgent：判断事件影响、有限传播和社交流程。
12. RelationshipDynamicsAgent / MultiDimensionalStateAgent：更新关系惯性和多维状态。
13. Reporter：整理用户可读日志。
14. 推进虚拟时间。
15. 睡眠、生理时钟、移动、地点影响、多维状态联动。
16. 生命状态判定。
17. 如果跨到 0 点，执行每日结构结算、明日计划、自我叙事、记忆衰退。
18. 自动保存当前存档。

简化流程：

```text
天气/地点事件
-> 地点今日重点/地点链
-> 地点运行态
-> 未完成过程检查
-> Scheduler 选人
-> AgentAction 行动
-> TimePassage 判断时间消耗
-> WorldGuard 本地硬审查
-> 状态落地
-> 事件影响/信息传播/社交流程
-> 关系惯性/多维状态
-> 时间推进
-> 生理/地点/多维联动
-> 生命状态判定
-> 0 点日结
-> 自动保存
```

如果全部角色死亡，系统会自动停止演算：

- 暂停运行
- 清空行动队列
- 不再推进时间
- 不再调用 AI
- 状态栏显示“全员死亡”

## AI 调用模块

当前 AI 调用分为多个任务类型：

- `worldSetupAgent`：根据用户设定补全初始小镇和角色。
- `scheduler`：选择哪些角色应该行动。
- `agentAction`：为单个角色生成具体小行动。
- `reporter`：把已发生记录整理成日志。
- `dailyPlanner`：每天 0 点生成明日动态计划。
- `weatherAgent`：生成天气观测、6 小时报告和 7 天趋势。
- `timeDecayAgent`：在确定性生理时钟后做因人而异微调。
- `locationEventAgent`：生成地点内部小事件。
- `locationInstitutionAgent`：生成地点制度、开放时段、课程、坐诊、营业和住宅作息规则。
- `locationDailyAgent`：生成地点今日重点、风险、岗位需求和公开提示。
- `locationChainAgent`：维护地点内部连续事件链，例如考试、候诊、缺货、停业、排班、维修、安全隐患。
- `locationRuntimeAgent`：判断地点此刻运行态，例如是否有工作人员、是否拥挤、哪些服务被阻塞。
- `processManagerAgent`：管理未完成的跨回合行动过程。
- `timePassageAgent`：判断行动消耗多少虚拟分钟、是否完成、剩余时间如何自然流逝。
- `eventImpactAgent`：判断已发生事件牵动哪些角色。
- `informationPropagationAgent`：判断信息如何通过同地点、家庭、同学、同事、邻居等有限渠道传播。
- `socialProcessAgent`：维护误会、冲突、隐瞒、澄清、道歉、和解、回避等社交流程。
- `relationshipDynamicsAgent`：判断关系的慢变量趋势和小幅漂移。
- `socialPatternAgent`：判断家庭、群体、关系对中的长期社会模式。
- `obligationAgent`：从行动结果中抽取承诺/任务债务。
- `stateSettlementAgent`：根据已发生行动提出需求、情绪、记忆、关系、地点状态补丁建议。
- `multiDimensionalStateAgent`：统一结算情绪、需求、关系、目标、地点状态。
- `selfNarrativeAgent`：每天 0 点更新角色自我叙事。
- `personalityConsistencyAgent`：每天 0 点更新人格锚点和稳定性。

设置支持：

- 默认模型
- 每类 Agent 单独模型
- 每个角色单独模型
- 多 API Key 分流
- 每 Key 最大并发
- 每轮最大 Action 数
- 自动重试，临时上游错误会按固定 300ms 重试
- 调用日志和状态栏

有效并发上限：

```text
有效并发 = Key 数量 x 单 Key 并发上限
```

例如 2 个 Key、每 Key 20 并发，则有效上限为 40。

## 角色系统

### 基础状态

每个角色包含：

- `id`
- `name`
- `job`
- `position`
- `currentTask`
- `energy`
- `sleepQuality`
- `sleepWindow`
- `dailyPlan`
- `household`
- `relations`
- `relationshipMatrix`
- `memory`
- `knownFacts`
- `lifeStatus`

### 年龄系统

年龄不是简单分类，而是真实数值：

- `ageYears`：年龄岁数
- `ageDays`：当前岁数内已过天数
- `ageStage`：由年龄自动推导

阶段规则：

- 13 岁以下：儿童
- 13-19 岁：青少年
- 20-64 岁：成年人
- 65 岁及以上：老人

每个虚拟日增加 1 个 `ageDays`。满 365 天后 `ageYears + 1`，并重新计算阶段。

年龄会影响：

- 饱腹消耗
- 清洁消耗
- 健康脆弱度
- 社交需求
- 责任压力
- 抗压稳定
- 舒适敏感
- 安全敏感
- 恢复速度
- 濒危到死亡的风险

### 多维情绪

当前角色拥有多维情绪：

- 开心
- 焦虑
- 愤怒
- 悲伤
- 疲惫
- 孤独
- 希望
- 平静
- 好奇

情绪会影响：

- 行为压力
- 记忆层级
- 关系反应
- 是否求助
- 是否回避
- 是否冲动
- 是否坚持当前职责
- 需求消耗或恢复

### 需求系统

需求采用养成游戏式状态条：

- `100` 表示状态充足
- `0` 表示危险终点
- 正数 `needDelta` 表示恢复/满足
- 负数 `needDelta` 表示消耗/恶化

当前需求包括：

- 饱腹
- 清洁
- 健康
- 社交满足
- 职责完成感
- 抗压稳定
- 舒适
- 安全

需求不是直接决定行为，而是经过时间、地点、身份、事件、情绪和性格调制。

例子：

```text
学生上课时饱腹低
-> 不能直接离开去早餐店
-> 冲动高：可能分心、小动作、焦虑
-> 忍耐高：继续听课，等下课处理
-> 健康很差：可以请假去诊所
```

### 多层需求联动

需求之间不是一层关系，而是多层耦合：

- 饱腹低会拖累健康、舒适、抗压。
- 清洁低会拖累健康、舒适、社交。
- 健康低会拖累责任、舒适、抗压。
- 社交低会增加孤独，降低抗压。
- 责任低会产生拖延，影响抗压和关系。
- 抗压低会让同样的小事更容易变成回避、冲突或求助。
- 安全低会提高焦虑，并拖累舒适和健康。
- 多项低状态叠加会产生系统性恶化。

### 情绪调制

角色还有一组稳定性格/当前心情调制值：

- 忍耐
- 冲动
- 求助
- 回避
- 坚持
- 社交开放

这些值由基础性格、多维情绪、需求状态共同计算。

同样的需求，在不同角色身上会表现不同：

```text
忍耐高：先撑住
冲动高：分心、小动作、打断节奏
求助高：告诉老师、家人、同事
回避高：沉默、离开风险点
坚持高：继续职责
社交开放高：主动交流
```

### 人格核心

当前版本新增 `identityCore`，用来让角色更稳定地“像自己”。

`identityCore` 包含：

- `values`：长期重视什么。
- `fears`：稳定担心什么。
- `habits`：遇事的惯性处理方式。
- `biases`：0-100 的人格偏向数值。

`biases` 当前包括：

- `dutyFirst`：职责优先。
- `riskAvoidance`：风险回避。
- `askForHelp`：求助倾向。
- `familyAttachment`：家庭牵引。
- `conflictAvoidance`：冲突回避。
- `statusConcern`：体面/评价压力。

这些数值会进入本地 `moodModulation` 和 `emotionBehaviorProfile`，影响 Scheduler 和 AgentAction 的倾向。  
例如同样是健康低：

```text
求助倾向高 -> 更可能找家人、老师、医生说明
职责优先高 -> 可能先尝试撑住职责
风险回避高 -> 更可能避险或去诊所
冲突回避高 -> 更可能沉默、延后对质或小心澄清
```

每天 0 点 `PersonalityConsistencyAgent` 可以小幅更新人格锚点和 `identityBiases`，但本地系统会限制为慢变化，避免角色每天变成另一个人。

## 生命状态和死亡系统

角色生命状态包括：

- `alive`：存活
- `critical`：濒危求助中
- `dead`：已死亡

触发濒危的情况：

- 健康归零
- 安全归零
- 饱腹归零
- 抗压归零
- 舒适归零
- 清洁归零

死亡不是单纯“某个数值到 0 立刻删除”，而是由多因素判定：

- 安全归零且处在危险地点或紧急事件中。
- 健康和安全同时归零。
- 健康归零后持续未救治。
- 饱腹归零并长期拖垮健康。
- 多个生命相关状态同时崩溃。
- 年龄会影响风险，老人更脆弱，儿童安全风险更敏感。

死亡后：

- 角色保留在存档和列表中。
- 不再参与 Scheduler。
- 不再执行行动。
- 不再睡眠、移动、生理衰减。
- 地图和列表中变灰。
- 如果全员死亡，演算自动停止。

## 地点 Agent

地点不是背景板。每个地点有自己的 Agent 状态：

- 压力
- 士气
- 负载
- 清洁
- 安全
- 当日重点
- 本地点相关承诺
- 本地点局部事件
- 当前在场人员
- 当前工作人员
- 是否有工作人员：`hasStaff`

当前地点包括：

- 广场
- 早餐店
- 学校
- 诊所
- 小卖部
- 居民楼
- 河边
- 镇政府

地点状态会影响角色：

- 压力高：角色更焦虑，职责状态消耗更快。
- 清洁低：清洁、舒适、健康受影响。
- 安全低：安全和焦虑受影响。
- 负载高：排队、候诊、服务压力增加。
- 天气差：移动变慢，河边风险上升。

### 地点连续事件链

`LocationChainAgent` 负责让地点内部事件具有连续性，而不是每轮随机刷新。

地点链包括：

- `place`：发生在哪个真实地点。
- `type`：学校事件、诊所流程、商店流程、住宅问题、天气问题、维修、安全隐患等。
- `stage`：`seed` / `noticed` / `active` / `handled` / `faded`。
- `status`：`open` / `resolved` / `faded`。
- `visibleTo`：当前真实看见或听见这条地点链的角色。
- `responsibleRoles`：制度上应该处理此事的岗位类型。
- `effects`：对地点服务、人流、情绪压力的影响。
- `history`：地点链的短历史。

例子：

```text
学校今日有小测
-> LocationDailyAgent 生成今日重点
-> LocationChainAgent 建立 school_event 链
-> LocationRuntimeAgent 让上课/考试时段影响地点运行
-> Scheduler 更可能调度相关学生、老师
-> AgentAction 只能让角色基于自己知道的信息反应
```

地点链不能创造隐藏工作人员，也不能直接替角色完成行动。它只提供地点层面的连续约束。

### 地点工作人员约束

地点不会自动拥有隐形 NPC。

系统会根据真实在场角色判断工作人员：

- 早餐店：早餐店老板、帮工、服务员、店员等职业才算工作人员。
- 小卖部：小卖部老板、售货、店员等职业才算工作人员。
- 诊所：医生、护士等职业才算工作人员。
- 学校：老师、校长、教务等职业才算工作人员。

如果某地点 `hasStaff=false`：

- AI 不能写店员、老板、服务员、收银员、医生、护士、老师等未在场工作人员。
- 只能写无人值守、自助、等待、柜台空着、老板不在、离开等情况。
- 前端落地时会做兜底过滤，防止 AI 仍然编出不存在的店员。

例子：

```text
早餐店只有一个买家，没有早餐店老板或帮工
-> hasStaff=false
-> 不能写“店员接单”
-> 应写“柜台无人，只能等待、自助查看或离开”
```

## 天气和日期

小镇有日期系统：

- 公历日期
- 星期
- 农历显示
- 季节
- 节气

WeatherAgent 会生成：

- 当前观测天气
- 未来 6 小时报告
- 1 天预测，可信度 50-85%
- 7 天趋势，可信度 10-50%
- 天气影响

天气会影响：

- 移动时间
- 地点安全
- 河边风险
- 疲惫
- 焦虑
- 舒适
- 地点 Agent 日结

## 承诺/任务债务系统

承诺是跨天追踪的任务债务。

承诺包含：

- 责任人
- 对象
- 地点
- 截止日
- 压力
- 情绪权重
- 相关需求
- 知道这件事的人
- 历史记录

承诺会影响：

- 职责完成感
- 社交满足
- 抗压稳定
- 焦虑
- 平静
- 关系事件链
- 长期记忆

承诺不会每个 tick 随机加压，而是在每天 0 点跨天结算。

## 记忆系统

记忆分层：

- `short`：短期记忆
- `long`：长期记忆
- `emotional`：情绪记忆
- `secret`：秘密
- `rumor`：传闻

记忆每天衰退，而不是每个 tick 衰退：

- 普通短期记忆衰退快。
- 传闻衰退快。
- 情绪记忆衰退较慢。
- 长期记忆衰退很慢。
- 秘密较稳定，但也会慢慢淡化。

普通小事不会全部进入长期记忆。

## 长期目标和自我叙事

每个角色有长期目标。

例子：

- 早餐店老板：照顾熟客，保住早高峰秩序。
- 学生：稳定完成学业和作息。
- 老师：照看学生状态和课堂秩序。
- 医生：维持诊所运转和复诊跟进。
- 家人：兑现修书包、复诊、照看老人等承诺。

长期目标会受到：

- 行动结果
- 承诺是否拖延
- 需求状态
- 情绪状态
- 关系变化
- 每日 0 点结算

SelfNarrativeAgent 每天 0 点更新角色如何解释自己昨天的行为，以及为什么继续坚持、回避或改变。

## 知识边界

每个角色不能全知。

角色能知道的信息来源包括：

- 自己的记忆
- 自己的已知事实
- 当前地点可见事件
- 同地点观察
- 家庭同步
- 直接交谈
- 知识传播
- 广播信息

角色不能直接知道：

- 全镇日志
- 别人的完整记忆
- 未公开承诺
- 未公开天气预报
- 远处发生但没人告诉自己的事
- 不在场的工作人员行为

当前系统通过 `visibleKnowledge`、`currentLocation.population`、`knownFacts` 和输出校验共同限制 AI。

## WorldGuard 本地审查

`WorldGuard` 是本地硬审查器，不调用 AI。它的作用是把“AI 觉得合理”再压回“世界里真的能发生”。

当前审查范围：

- 地点 id 必须真实存在。
- 移动中不能开启第二段移动。
- 没有真实工作人员时，不能写店员、老板、医生、护士、老师等隐藏 NPC。
- 独处时不能写有人气、顾客、路人、闲聊、社交缓冲。
- AgentAction 不能生成死亡、复活或死亡判定。
- 不能写全镇广播、所有人都知道、大家都听说。
- TimePassage 判断未完成时，不能写已经到达、已经买到、已经看完、已经解决。
- 关系变化对象必须是该角色知道或可接触的人。
- newEvents.knownBy 只能是该角色、同地点可见角色、家人同步或可解释传播对象。
- emotionDelta 只能保留合法情绪维度，并做限幅。
- 越权记忆会被丢弃。

处理方式：

```text
合法 -> 保留
部分越权 -> 删除越权字段
严重越权 -> 降级为 observe/wait
```

这条规则保证 AI 可以提供生活判断，但不能最终决定世界事实。

## 社交流程链

`SocialProcessAgent` 负责把已发生事件、有限传播和关系惯性组织成社交流程。

它处理的不是“全镇剧情”，而是局部社会反应：

- 误会
- 冲突
- 隐瞒
- 道歉
- 澄清
- 和解
- 回避

社交流程包含：

- `participants`：直接参与者。
- `knownBy`：当前知道这件事或这个流程的人。
- `hiddenFrom`：暂时不知道真相或被隐瞒的人。
- `truth`：系统层面确认的简短事实。
- `beliefs`：各角色当前相信的版本。
- `stage`：`seed` / `noticed` / `tension` / `confront` / `clarify` / `reconcile` / `fade`。
- `tension`：当前张力。
- `nextPossibleActions`：后续可能方向，但不是已经发生的行动。

例子：

```text
一个学生在诊所看起来不舒服
-> EventImpactAgent 判断直接影响谁
-> InformationPropagationAgent 判断谁通过同地点、家人、老师知道
-> SocialProcessAgent 形成“担心/误会/隐瞒/澄清”流程
-> Scheduler 后续更可能调度家人、老师或同学关心
-> AgentAction 仍然只能基于该角色实际知道的信息行动
```

如果发生死亡、严重冲突、迟到、失约、疾病、请假、争吵、缺货、停业等事件，也应先经过事件影响和信息传播，再进入社交流程。角色不会因为全局日志而凭空知道。

### 阶段推进硬规则

社交流程现在不能一轮直接从误会跳到和解。  
本地系统会限制阶段：

```text
seed
-> noticed
-> tension
-> confront
-> clarify
-> reconcile
-> fade
```

限制规则：

- 新流程最多从 `seed` / `noticed` / `tension` 开始。
- 已有流程每次最多推进一小步。
- 距离上次更新太近时，不允许快速推进。
- `confront` 至少需要两个真实参与者。
- `clarify` 需要解释、澄清、问清等证据。
- `reconcile` 需要道歉、和解、说开、澄清等证据。
- `resolved` 只有在阶段和证据都支持时才允许。
- 投递到角色事件队列时，会按 `knownBy` / `hiddenFrom` 隐藏真相；不知道真相的人只会看到自己的 belief 版本。

## 每日 0 点结算

每天跨到 00:00 时，系统会执行结构结算：

1. 年龄增长 1 天。
2. 承诺/任务债务跨天结算。
3. 地点 Agent 日结。
4. 长期目标和性格稳定性微调。
5. 关系矩阵日结。
6. DailyPlanner 生成明日动态计划。
7. SelfNarrativeAgent 更新自我叙事。
8. 每日记忆衰退。

流程：

```text
跨天到 00:00
-> 年龄增长
-> 结构结算
-> 明日计划
-> 自我叙事
-> 记忆衰退
```

这样明天不是简单重复今天，而是带着昨天留下的承诺、压力、目标、情绪、地点状态和关系继续运行。

## 当前真实度评估

当前版本真实度评估为：

```text
78 / 100
```

它已经不是简单“角色每轮说一句话”的聊天模拟，而是有生活规则、地点约束、知识边界、事件传播和跨天记忆的小镇沙盘。

当前已经具备：

- 时间约束
- 地点约束
- 工作人员存在性约束
- 角色身份
- 年龄成长
- 需求驱动
- 多维情绪
- 情绪调制
- 人格核心和人格偏向
- 天气影响
- 地点 Agent
- 地点连续事件链
- 承诺债务
- 长期目标
- 自我叙事
- 分层记忆和每日衰退
- 知识边界
- WorldGuard 本地硬审查

- 有限信息传播
- 社交流程链
- 社交流程阶段推进规则
- 关系慢变量

- 生命状态和死亡
- 全员死亡停止演算
- 每日结构结算
- 存档和配置文件

真实感主要来自这些链路：

```text
地点产生约束
-> 角色只基于自己知道的信息行动
-> 行动消耗虚拟时间
-> 事件影响真实在场/相关的人
-> 信息有限传播
-> 形成误会、冲突、澄清、和解等社交流程
-> 关系和记忆跨天沉淀
-> 第二天的计划和人格倾向被轻微改变
```

它现在比较像“有真实模拟骨架的小镇 Demo”。但还不是完整真实社会，原因是长期生活积累、地点服务流程、职业组织、经济/物品、公共制度和社会网络密度还不够。

## 主要不足

当前仍需要增强：

- 地点内部事件链已经有了，但学校、诊所、小卖部还需要更稳定的服务流程、排班、预约、补货和异常处理。
- 社交流程链已经有了，但冲突、误会、隐瞒、澄清、和解还需要更多跨回合自然发酵。
- 承诺仍需要更自然地大量生成和消解。
- 关系矩阵虽然已有多维和慢变量，但还需要更强的长期社会网络影响，例如熟人圈、阶层、口碑、嫉妒、依赖和排斥。
- 知识边界已经加强，但 AI 偶尔仍可能写出角色不该知道的内容，需要更多本地审查和纠错。
- 行动颗粒度虽然有跨回合过程，但仍有不少行动偏粗，需要更多“准备、路上、等待、执行、反馈”的连续过程。
- 还没有物品模块和金钱模块。
- 没有真正的店面经营库存、服务流程和排班系统。
- 没有职业组织规则，例如学校班级、诊所预约、店铺营业制度。
- 死亡和重大危机的社会影响已经能进入事件传播和社交流程，但家人长期反应、公共记录、关系链断裂还较弱。

## 下一步建议

建议按以下顺序继续：

1. 地点排班/工作人员系统  
   让店铺、学校、诊所知道谁该上班，谁缺勤，谁临时代班。

2. 地点服务流程  
   早餐店买早餐、诊所看病、学校上课、小卖部购物应有多步流程。

3. 职业组织规则  
   学校有班级和课程，诊所有预约和候诊，店铺有开关门和备货。

4. 死亡和重大危机的社会后果  
   家人、邻居、地点、公共事件都应被影响。

5. 更硬的知识审查  
   输出落地前检查角色是否知道某人、某事、某地点状态。

6. 物品模块  
   先加少量核心物品：早餐、药、书包、作业本、钥匙、伞。

7. 金钱模块  
   后加，避免过早让系统复杂化。

目标结构：

```text
地点产生真实约束
-> 角色基于自己知道的事反应
-> 关系决定反应方式
-> 行动生成承诺/误会/任务
-> 承诺跨天发酵
-> 0 点形成目标、情绪和明日倾向
-> 第二天继续生活
```

## 开发原则

后续每添加一个模块，都需要明确回答：

1. 它影响哪些需求？
2. 它影响哪些多维情绪？
3. 它是否受情绪调制影响？
4. 它是否受地点、时间、身份约束？
5. 它是否需要进入知识边界？
6. 它是否需要 0 点结算？
7. 它是否会生成承诺、关系变化或记忆？
8. 它是否会影响生命状态？
9. 它是否需要工作人员、物品或金钱前置条件？

只有回答清楚这些问题，模块才算真正接入小镇，而不是单独的装饰功能。

## 生命与生理优化补充

当前版本把生命系统从“硬核生存”调整为“小镇生活”：

- 基础衰减放缓：饱腹、清洁、责任、舒适、安全不会按小时快速清零。
- 多维联动保留但降强度：饱腹低会拖累健康，清洁低会影响健康和社交，压力低会拖累健康，但不会几轮内连锁崩溃。
- 死亡阈值放宽：健康归零后按半天到一天以上计算，长期饥饿按多天计算；安全事故仍更危险，但也需要持续濒危。
- 基础求生兜底：吃饭时间在家、早餐店或有工作人员的小卖部时，角色会自动最低限度补充食物；人在诊所且健康极低时会获得基础救治。
- 饥饿紧急调度：饱腹极低会进入 Scheduler 的紧急候选，避免只等待普通行动。
- AI 扣健康限幅：TimeDecayAgent 和 MultiDimensionalStateAgent 单次不能把健康大幅打穿。
